# Adopt the `DESIGN.md` format spec + lock axe baseline

> Research-phase proposal generated by `/ccg:spec-research` on 2026-05-25. Source plan:
> `.claude/plan/google-design-incorporation.md` (verbatim user request from 2026-05-21).
> Audit performed against the upstream spec at
> [`google-labs-code/design.md`](https://github.com/google-labs-code/design.md) (commit cloned
> to `~/_work/repos/google-labs-code-design.md/` on 2026-05-25; `version: alpha`).

## Why

LifeOS already has a strong, opinionated design system: tokens in `colors_and_type.css` (60+ CSS variables across 8 token families) and long-form rationale in `design-system-reference/README.md`. What it does not have is an **agent-readable canonical file** that AI coding agents can consume without re-deriving the system from prose every session. Multiple agents (Claude Code, Codex, Gemini, plus future tooling) work in this repo daily; each one currently re-builds a mental model from `README.md` + `colors_and_type.css`, and they disagree about the system in subtle ways (radius scale naming, which colors are "brand" vs "status", what the typography levels actually are).

Google Labs' [`design.md` format spec](https://github.com/google-labs-code/design.md) (alpha, Apache-2.0 licensed) is **exactly the file format we are missing**: YAML front matter holds the normative tokens, an 8-section markdown body holds the rationale, and a CLI (`@google/design.md`) lints the file against 8 structural + accessibility rules — including `contrast-ratio` (WCAG AA 4.5:1, token-level). Crucially, it is a *format*, not a UI library, not Material Design, not a CSS framework. Adopting it costs us no visual identity change. It costs us one new file (`DESIGN.md`), one devDependency, and one new lint script.

The second half of this change closes a long-standing gate gap. The loop-closure documented "0 axe violations across 9 surfaces" — but `grep -rn axe` across the repo returns zero matches. That 0-violation baseline was achieved with a manual dev-tools sweep and is not regression-protected. Component-level automated axe via `vitest-axe` (chosen scope, ~30 specs) turns the existing baseline into an enforced contract: any future change that introduces an a11y regression on a covered surface will turn a test red instead of slipping past until the next manual sweep.

Together the two halves are coherent: DESIGN.md gives agents (and humans) a machine-readable contract for *what the design is*; component-level axe gives the suite an automated contract for *whether the rendered output honors accessibility*. The `contrast-ratio` lint rule and the axe `color-contrast` rule are complementary — the former checks the tokens as authored, the latter checks the DOM as rendered.

## What Changes

### Part A — `DESIGN.md` format adoption

- Create `DESIGN.md` at the repo root. YAML front matter contains the normative tokens. Markdown body uses the canonical 8-section order (Overview, Colors, Typography, Layout, Elevation & Depth, Shapes, Components, Do's and Don'ts).
- Add `@google/design.md` as a devDependency (CLI + linter library only, never imported by the runtime). Pin to the exact version published on npm at adoption time — the format is at `version: alpha` and may evolve.
- Add `bun run design:lint` (`@google/design.md lint DESIGN.md`) — gates on 0 errors. Wired into `bun run check` (precommit) and CI.
- Add `bun run design:diff` — detects token-level regressions vs. the previous-release snapshot at `design-system-reference/exports/DESIGN.previous.md`. **The upstream CLI takes file paths, not git rev syntax** (codex finding: `DESIGN.md@HEAD~1` would be passed as a literal filename and fail); a small Node wrapper at `scripts/design-diff.mjs` shells `git show HEAD~1:DESIGN.md` into a tempfile, then calls the CLI. Advisory locally, hard gate in CI; falls back to no-op if no previous version exists.
- Add `bun run design:export:dtcg` (DTCG `tokens.json`) and `bun run design:export:tailwind` (Tailwind v4 `@theme` CSS) — output artifacts checked in under `design-system-reference/exports/` for cross-tool consumption (Figma, Tailwind users, downstream apps).
- `design-system-reference/README.md` stays as the long-form brand guide. Add a short header pointer: *"Normative tokens live in `DESIGN.md` at the repo root. This document covers identity, voice, asset attribution, and component patterns that don't fit the token schema."* No content is moved or duplicated — the README keeps its identity prose; DESIGN.md keeps the numbers.

### Part B — Lock the 0-violation axe baseline

- Add `vitest-axe` as a devDependency. Wire its `toHaveNoViolations` matcher in `tests/setup.ts` (currently absent — create it; add `setupFiles` entry in `vitest.config.ts`).
- Add component-level axe specs (~30 specs, per the chosen scope) under `tests/a11y/`. Coverage targets:
  - Every dedicated view at idle: Dashboard, Lights, Calendar, Files, Health, IoT, ContactsView (workspace), ContactsView (aggregator), SettingsView, OpenPencilEditor, N8nFlowView (11 surfaces).
  - Every interactive overlay open + closed: CommandPalette, KeyboardHelp, NotificationsDrawer, ToastContainer (4 components × 2 states = 8).
  - Every stateful component variant: Button (idle / hover / disabled / focus-visible), Sidebar (expanded / collapsed), AIAvatar (hidden / visible / chat-open), TelemetryWidget (loading / loaded / error) — ~10 additional.
- Each spec renders the component in happy-dom (existing test env), waits for any async, runs `axe.run()`, asserts 0 violations against `wcag2a, wcag2aa, wcag21aa`.
- **Pseudo-class limitation (codex finding):** happy-dom does not fully compute `:hover` / `:focus-visible` / `:active`, so axe's `color-contrast` rule against pseudo-state styles is unreliable in this environment. Coverage is for **structural a11y** (roles, names, labels, landmarks, contrast on rendered-default state). Hover/focus contrast claims, if needed, would require a follow-up Playwright lane — explicitly out of scope here.
- Add `bun run test:a11y` script that filters to `tests/a11y/`. **Update `vitest.config.ts` `include` to exclude `tests/a11y/**` so `bun run test` stays at 194 specs unchanged** (codex finding: default include `tests/**/*.spec.{js,ts}` would pull a11y specs into the main run by default).

### Part C — CI / commit-hook wiring

- Update `package.json` `scripts.check` (or create it if absent) to run, in order: `vue-tsc --noEmit`, `bun run test`, `bun run test:a11y`, `bun run design:lint`. Single command for local pre-flight.
- No GitHub Actions changes proposed in this OPSX change (the repo has no `.github/workflows/` today — that's a separate change).

## Capabilities

### New Capabilities

- `design-system-contract`: A normative `DESIGN.md` at the repo root, valid against the upstream `@google/design.md` linter, containing the LifeOS token universe (colors / typography / rounded / spacing / components) plus the 8-section prose. Owned by `DESIGN.md`, `design-system-reference/README.md` (companion prose), and `design-system-reference/exports/` (DTCG + Tailwind exports).
- `design-system-lint`: Token-level structural and contrast validation via `@google/design.md lint`. Owns the 8 rules: `broken-ref` (error), `missing-primary` (warning), `contrast-ratio` (warning, WCAG AA), `orphaned-tokens` (warning), `token-summary` (info), `missing-sections` (info), `missing-typography` (warning), `section-order` (warning). Surface: `bun run design:lint`.
- `a11y-regression-suite`: Component-level axe-core specs that lock the 0-violation baseline as a test-runtime contract. Surface: `bun run test:a11y`, `tests/a11y/*.spec.ts`, `tests/setup.ts` (axe matcher wiring).
- `design-token-exports`: DTCG `tokens.json` and Tailwind v4 `@theme` CSS generated from `DESIGN.md`. Owned by `design-system-reference/exports/{tokens.json, theme.css}`, regenerated by `bun run design:export:*`.

### Modified Capabilities

None of the existing OPSX specs (`account-persistence`, `local-storage`, `mempalace-mirror`, `ruvector-mirror`) are touched. The change adds capabilities; it does not mutate prior contracts. `colors_and_type.css` is read-only in this change — DESIGN.md mirrors it but is not generated from it (initial draft is hand-authored; subsequent edits must update both, enforced by the `broken-ref` lint catching any `var(--*)` reference whose token doesn't exist in YAML).

## Constraints discovered

### Hard (cannot be violated)

1. **Color tokens in DESIGN.md YAML must be opaque `#hex` SRGB.** Per spec.md and the `Color` type definition. LifeOS has 14 `--tint-*` companions using `rgba()` — these stay as CSS-only helpers. The DESIGN.md `colors:` map contains only the opaque base colors. Where a component needs an alpha (e.g. tinted chip backgrounds), the raw `rgba(...)` literal is permitted **inside the `components:` map** (verified against `examples/atmospheric-glass/DESIGN.md` which uses `rgba(255, 255, 255, 0.1)` in `glass-card-standard.backgroundColor`).
2. **The `--gradient-spiral` brand mark has no token type in the DESIGN.md spec.** Spec covers Color (`#hex`), Dimension (`px`/`em`/`rem`), and Typography only. Resolution: the gradient is documented in **prose** under `## Colors` ("The signature spiral gradient — cyan → purple → green at 135° — is the only chromatic moment...") and referenced as a raw value in the `components:` map where used (e.g. `button-primary.backgroundColor: "linear-gradient(135deg, #00D4FF 0%, #9B7BFF 50%, #00E676 100%)"`). The `broken-ref` lint will warn but not error on the raw value.
3. **No `openssl-sys` in the cargo graph.** Pre-existing constraint from `AGENTS.md` and the storage-foundation change. `@google/design.md` is a Node.js CLI — zero impact on the Rust graph. Verified by `cargo tree -p lifeos-core --features storage | grep openssl-sys` staying empty.
4. **ESP32 / `no_std` isolation must hold.** `cargo check -p lifeos-core --no-default-features` must continue to pass. This change is entirely JS-side; the Rust crate is untouched. No risk.
5. **`bun test` must stay at ≥194 passing.** Adding ~30 a11y specs grows the count to ~224. Any failure in an a11y spec OR a regression in an existing spec blocks the change. Per `AGENTS.md` "Never commit without `bun test` passing".
6. **`bun run build` must stay green.** `vue-tsc --noEmit` + Vite production build. New devDeps and scripts must not break the build path.
7. **OpenPencil compatibility preserved.** Per `AGENTS.md`: "All changes must remain compatible with OpenPencil local editing." No changes to `App.vue` mounting gate or `useNav().pickSub(...)` flow. Axe specs for `OpenPencilEditor.vue` use the existing test harness.
8. **No new runtime dependencies.** `@google/design.md` and `vitest-axe` are devDependencies only. Bundle size impact on the shipped app is 0 bytes. The `manualChunks` config in `vite.config.ts` is untouched.
9. **`.ts` / `.js` sibling parity preserved.** Per `CLAUDE.md`: the dual-mode store, persistence whitelist, and resolve helpers have `.ts` and `.js` versions that must stay sibling-identical. This change touches neither — only `DESIGN.md`, `package.json` scripts, `tests/a11y/`, and `tests/setup.ts`.

### Soft (preferred, with rationale)

1. **Color naming follows spec convention** (`primary`/`secondary`/`tertiary`/`neutral`/`surface*`/`outline`/etc.), mapped to LifeOS identity in prose: `primary` = cyan (action/focus/link), `secondary` = purple (AI/thinking/memory), `tertiary` = green (success/accent/live), `neutral` = fg-1. CSS variables in `colors_and_type.css` keep their existing `--lifeos-*` names — no breaking rename. **Why**: maximises agent-readability (the spec's recommended names are what agents reading a DESIGN.md expect to see); the prose carries the LifeOS-specific identity story.
2. **Typography names follow spec convention** (`display-*`/`headline-*`/`body-*`/`label-*`/`mono-*`). Map to LifeOS `.t-*` utility classes: `display-stat` = `.t-stat`, `display-hero` = `.t-display`, `headline-lg/md/sm` = `.t-h1/h2/h3`, `title-md` = `.t-h4`, `label-caps` = `.t-section`, `body-md` = `.t-body`, `body-md-strong` = `.t-body-strong`, `label-sm` = `.t-meta`, `mono-sm` = `.t-code`. **Why**: same as above — agent-readability.
3. **Spacing scale uses `xs/sm/md/lg/xl/2xl/3xl/...` names** mirroring `--space-1` through `--space-16`. **Why**: spec's recommended scale.
4. **Rounded scale renames `--radius-capsule` → `full`** in the DESIGN.md map (CSS variable name unchanged). **Why**: `full` is the spec's recommended name for `9999px`.
5. **Components section covers design-system-level components only**, not every Vue SFC. Initial scope: Button (primary / secondary / ghost / icon), Card (standard / glow / stat), Input (text / search), MenuRow (idle / hover / active), Chip (filter / status), Badge (count / dot), KbdKey. **Why**: the spec's `components:` map is for token-level styling, not behavioral specification. Workspace-specific surfaces (Dashboard, LightsView, etc.) consume these via CSS variables and don't need their own token entries.
6. **`@google/design.md` version pinned exactly** (no `^` range) for the duration of the `alpha` phase. **Why**: spec README explicitly warns "expect changes to the format as it matures".

### Dependencies (cross-module ordering)

- `tests/setup.ts` creation (or augmentation) must land before any `tests/a11y/*.spec.ts` — the axe matcher wiring is global.
- `DESIGN.md` must exist before `bun run design:lint` is added to `bun run check` — otherwise local pre-flight breaks for everyone until the file lands.
- `design-system-reference/exports/` directory creation must precede the `design:export:*` script wiring.
- Order of patches in the implementation phase: (1) DESIGN.md draft + lint script, (2) verify lint passes with 0 errors, (3) axe matcher wiring + first 5 a11y specs as proof, (4) full ~30-spec suite, (5) export scripts + checked-in artifacts, (6) wire into `bun run check`, (7) update `design-system-reference/README.md` header pointer.

### Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `@google/design.md` schema changes mid-implementation (it's `alpha`) | Medium | High (re-author DESIGN.md) | Pin exact version; subscribe to repo releases; the format breaking would be visible at install time, not at runtime |
| Axe finds violations the manual sweep missed | Medium | Medium (extra fix work) | Treat first failures as discovery, not regression; fix inline before merging the suite; do not silence false positives without documenting |
| `contrast-ratio` lint warns on cyan-on-near-black (4.5:1 boundary case) | Medium | Low (warning, not error) | Spot-check cyan `#00D4FF` against `#0A0A0A` = ~12.1:1, passes AA; against `#1A1A1A` (card) = ~10.4:1, passes AA. Brand glow tints (`rgba(0,212,255,.14)` on `bg-2`) live in components map as raw values and the lint won't compute contrast against rgba — verify rendered contrast via axe instead |
| Vitest-axe slows the unit suite (~10s added for ~30 specs) | High | Low | Acceptable per chosen scope; CI runs `test` and `test:a11y` as parallel jobs; if local feedback loop becomes painful, split `bun run test` to exclude `a11y/` and require `bun run check` for full pre-flight |
| Sibling `.js` files (preview path) miss DESIGN.md guidance | Low | Low | No code change to `.ts`/`.js` siblings in this OPSX change — purely doc + test + tooling |
| Future contributor edits `colors_and_type.css` without updating `DESIGN.md` | Medium | Medium (drift) | Add a `## Synchronization` note to DESIGN.md + a CONTRIBUTING note; consider a follow-up OPSX change that auto-generates one from the other |
| `vitest-axe` matcher type collision with `@vue/test-utils` global types | Low | Low | Document type augmentation in `tests/setup.ts`; verified by `vue-tsc --noEmit` step |

- **Replacing Lucide icons** — explicitly preserved per `AGENTS.md`.
