This document is the durable capture of the supply-chain brainstorm so it is **not
lost during implementation**. It defines the data model, the three lanes, the
install taxonomy, the RuVector latest-source ruling, registry-key handling, and the
one law that prevents agents from looping.

---

## ⛔ THE NO-OPTIONAL LAW (read this first)

**There is no `optional`, `install-later`, `nice-to-have`, or `non-blocking` state.
Every row in the catalog is BUILT before DONE. "Optional" means *install*.**

An agent that treats any catalog row as deferrable, optional, or non-blocking has
entered a **failure loop** (a black hole of "I'll add it later" that never
converges). The word `optional` is therefore **banned as a classification value**.
The catalog validator fails if the token `optional` appears as a disposition.
Everything listed is required; the only thing that varies is *which lane* it lands in.

## 1. Data model — separate registry, one-way FK into the task graph

The supply chain is a **separate registry file**, not extra columns on the task graph.

- **File:** `supply_chain.catalog.csv` (source of truth), beside `task_graph.source.csv`.
  Generated → `generated/supply_chain.catalog.json` (mirrors the existing
  `*.source.csv → generated/*.json` pipeline; regenerated by the profile-owned Nushell).
- **Link direction:** each catalog row carries a `task_id` **foreign key INTO** the
  task graph. The task graph is **never** given supply-chain columns and is **never**
  edited by catalog churn. The reverse index (task → packages) is built **at
  generation time**, not stored on the task rows.

**Why not columns on the task graph:** many-to-many cardinality (one task provisions
many packages; one package is touched by many tasks), different lifecycles (a version
bump is upstream-driven, not work-driven), and the task CSV is pinned by brittle
byte-for-byte navigation + exact-length owner-contract tests — coupling would churn
both on every version bump.

## 2. Three lanes — ALL profile-owned or project-local-pinned (never system/global)

The profile owns the **entire** stack (blueprint Phase 4: "compile the entire stack
into fully static musl binaries, completely dropping the Nix store dependency").
The lane axis is **runtime shape**, not in-profile-vs-out.

| Lane | What | Build/run mechanism | Profile-owned? |
|---|---|---|---|
| `path` | agent CLIs & tools (claude, codex, grit, icm, weave, bun, bunx, rtk, nu, envctl, …) | flake output → yazelix runtime closure → `~/.nix-profile/bin` (or `toolbin`) | **yes** |
| `service` | **PostgreSQL + RuVector extension** | flake-declared package in the runtime closure; run as a **profile-owned daemon**; extension loaded via `shared_preload_libraries`; data dir **outside** the store | **yes (MUST — see §5)** |
| `app-dep` | libraries the app links — **ruvector** (`bun add`, npm latest) for the TS/Bun front-end; `ruvector-core`/`ruvllm` (cargo) for the Rust daemon | project-local pinned lockfile (bun / cargo); toolchain is profile-owned | library = project-local; toolchain = yes |

**The no-global / no-system-depth law reduces to:** everything is either
**yazelix-profile-owned** (`path`, `service`, and *every* toolchain) or
**project-local-pinned** (`app-dep`). Nothing is `apt`, system-depth, or `npm -g`.
`bun add ruvector` writes the app's own `node_modules` — **project-local ≠ global
install → compliant.**

> **ruvector spans two lanes** as different artifact forms of one project:
> `service` (the profile-owned PG extension) and `app-dep` (the npm/cargo library).
> That is one row per `(package × lane)`, both required.

## 3. The RuVector latest-source ruling (owner: use LATEST = npm)

The blueprint contains a real tension; the owner resolved it. Both are cited from
`Architecture Blueprint - LifeOS Core Foundation.md`:

- **Lines 87–91 — "Why use NPM releases first":** the CI "cross-compile[s] and
  push[es] updates directly to NPM the moment a new branch is merged," so **npm is
  newest and crates.io lags** ("Updating Crates.io … naturally lags behind this
  rapid 'stream of consciousness' development"). The TS/Bun front-end uses
  `bun add ruvector`.
- **Lines 93–113 — "napi-rs exhaust" / "Fall back on Crates.io":** for a *pure* Rust
  build, pull `ruvector-core`/`ruvllm`/`ruvector-sona` from crates.io and **"completely
  avoid cloning the main RuVector (meta-ruvector) GitHub repository,"** a "messy
  scratchpad" of napi-rs exhaust — **but this "might miss … bleeding-edge features."**

**Owner ruling (`DEC-YZXCONV-007`):**
- **npm latest WINS.** The latest npm build carries **200+ built-in AI agent tools**;
  an older resource yields only **~80–120**. Using anything but latest = fighting
  continuous refactoring.
- **crates.io = the blueprint's *purity* fallback** that itself admits it lags /
  misses features → **deprioritized**.
- **Local `meta-ruvector` + worktrees = reference only** (the blueprint's own
  "scratchpad / napi exhaust"; owner's ~60% stale read). Explore if/when needed.

## 4. Catalog schema (columns)

Two axes are kept **independent**: `source_registry` (where *latest* lives) vs
`install_method` (how nix/the app *builds* it) — because for ruvector the latest is
npm but the build is a flake/PG-extension (service) or a bun/cargo dep (app-dep).

| column | purpose | example |
|---|---|---|
| `pkg_id` | stable slug | `SC-rtk`, `SC-ruvector-ext`, `SC-ruvector-npm` |
| `name` | canonical name | `rtk`, `ruvector`, `@ruvnet/ruvector` |
| `kind` | what it is | `rust-bin` \| `rust-crate` \| `napi-npm` \| `pg-extension` \| `mcp-server` \| `plugin` |
| `lane` | runtime shape | `path` \| `service` \| `app-dep` |
| `source_registry` | where **latest** lives | `npm` \| `crates.io` \| `git` \| `local-path` |
| `source_locator` | the coordinate | `npmjs/@ruvnet/ruvector`, `/home/flexnetos/meta/src/meta-ruvector` |
| `pinned_version` | current pin | `latest` (npm dist-tag) / `0.8.0` |
| `install_method` | how it's built | `flake-crate` \| `flake-git` \| `flake-npm` \| `flake-pg-extension` \| `bun-add-latest` \| `cargo-dep` \| `yazelix-runtime` |
| `nix_output` | flake attr (path/service) | `packages.x86_64-linux.rtk` |
| `profile_owned` | in the yazelix profile? | `true` (path/service + toolchains) / `false` (app-dep library) |
| `classification` | **`required` only** — `optional` is illegal | `required` |
| `depends_on` | supply-chain deps (`;`-joined) | `SC-postgres` |
| `platform_variant` | musl caveat | `musl` \| `gnu` \| `any` |
| `registry_key_scope` | auth class (never the value) | `public` \| `npm-authed` \| `cratesio-authed` |
| `task_id` | owning task FK | `YZXCONV-034` |
| `proof_uri` | verification artifact | `proof_records/SC-rtk.proof.json` |
| `notes` | edge cases | "latest npm; crates.io lags; local = reference" |

## 5. `service` lane — the databases MUST be in the profile

Owner correction (2026-07-14): "**none of the databases are in yazelix nix-profile**
[today]. The **Databases MUST be IN the YAZELIX NIX-PROFILE** to pull this off."

So PostgreSQL **and** the RuVector extension are packaged **the yazelix way** — a
flake output **and** a declared runtime-closure package (as `rtk` is) — so the single
profile owns the whole stack:
- `postgres` = profile-provided engine; `ruvector` = profile-provided extension `.so`.
- Loaded via `shared_preload_libraries`; `CREATE EXTENSION ruvector` succeeds.
- Data dir lives **outside** the nix store (never committed; never in `/nix/store`).
- **No** system/apt postgres, **no** global install.

This is tracked as **`YZXCONV-035`** (Blocked until the catalog service rows + flake
packaging land).

---

## 6. Registry keys — via the broker, never in the store

npm + crates.io keys authenticate scoped/private fetches. **Never bake them into the
flake** (they'd land world-readable in `/nix/store`). Keys come from **secretd →
secretctl → env at build time** (`nix build --impure` / netrc / `.npmrc` /
`CARGO_REGISTRIES_*_TOKEN`). The catalog records only `registry_key_scope` (the key
*class* needed), never a value.
